# SOP (Standard Operating Procedure)

**An SOP is a reusable workflow that captures the decisions, steps, and checkpoints for a recurring task — so you build it once, run it repeatedly, and improve it over time.**

When you first start delegating to AI, every task is a new brief. You describe the outcome, set constraints, and hit go. That works — but it means you're inventing the approach from scratch every time. An SOP is what happens when a delegation proves itself and you decide to keep it.

Think of an SOP as delegation that has graduated. Instead of "I need you to write a weekly status report — here's the format, here are the inputs, here's what to check," you say "Run the weekly status report SOP." The agent knows the format, the inputs, the checks. You've encoded the pattern once and use it forever. That's the difference between delegating a task and building a system.

## Why It Matters for Moving Beyond Prompting

The operator asks: "What prompt should I use for this?" The orchestrator asks: "What brief do I need to write?" The architect asks: "Where does this belong in my SOP library?"

This is the progression that the EvoSOP research makes concrete. Agents can evolve from one-off atomic actions into reusable SOPs through a lifecycle: actions are captured, similar actions are merged into candidate procedures, the procedures are tested for reliability, and underperforming ones are pruned. Each iteration makes the library stronger. Each new task that gets captured adds to the foundation.

For you, the practical version is simpler: every time you delegate a task that you'll need again, save the brief as a named SOP. It can be a single markdown file, a saved template in your AI tool, or a document in your knowledge base. What matters isn't the format — it's the shift from "I'll figure out the prompt when I need it" to "I already know how to do this, and the system remembers."

## What Goes Into an SOP

A good AI SOP has four parts:

1. **Inputs** — What the agent needs before it starts (files, links, context, dates)
2. **Outputs** — What finished work looks like (format, length, structure)
3. **Constraints** — What it must NOT do and what tradeoffs to prioritize
4. **Checkpoints** — Where a human reviews and where the agent can keep going

The difference between an SOP and a prompt is the **checkpoints**. A prompt is fire-and-forget. An SOP has review points built in — places where you verify direction before the agent continues. Those checkpoints are what make delegation safe enough to be automatic.

## Try This

Pick one task you delegated successfully this week. Open a blank document and write the four-part SOP: Inputs, Outputs, Constraints, Checkpoints. It doesn't need to be long — a paragraph per section is fine. Name it something you'll remember. Next time that task comes up, reference the SOP instead of writing a new brief. Notice: did it save time? Did it produce a more consistent result? That's the compounding effect of reusable delegation.
